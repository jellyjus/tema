🔥 Шаг 3. Задание связей (relations)
🎯 Цель

Создать систему, в которой:

    рынки связаны логически

    связи можно задавать, менять и тестировать

    модель получает осмысленные входные данные

1. Что такое “связь” в системе

Связь — это:

    влияние одного события (рынка) на вероятность другого

Формально:

source_market → target_market

С параметрами:

    тип влияния

    сила влияния (вес)

2. Что нужно реализовать (технически)

Программист должен:
2.1 Сделать возможность добавления связей

Минимум:

    через БД (ручное заполнение)

    или простой интерфейс / JSON

2.2 Формат одной связи

source_market_id  
target_market_id  
relation_type (positive / negative)  
weight (0–1)

2.3 Возможность:

    добавлять связь

    редактировать

    удалять

3. Логика использования связей

Для одного target-рынка:

может быть НЕСКОЛЬКО source-рынков

👉 это основа многофакторной модели
4. Методология (самое важное — для вас)

Вот где большинство ошибается.
🔥 Правило 1. Связь должна быть причинной, а не “похожей”

❌ Плохо:

BTC ↔️ ETH (просто потому что оба растут)

✅ Хорошо:

Liquidity → BTC

🔥 Правило 2. Связь должна иметь направление

Всегда:

что влияет → на что влияет

Не:

A ↔️ B

А:

A → B

🔥 Правило 3. Не использовать “очевидные отражения”

❌ Плохо:

BTC up → BTC down

👉 это один и тот же рынок
🔥 Правило 4. Связь должна быть интерпретируемой

Ты должен уметь сказать:

    “Если A растёт, то B растёт/падает потому что …”

Если не можешь объяснить — связь плохая
5. Типы связей
   🟢 Positive

A ↑ → B ↑

Пример:

Liquidity ↑ → BTC ↑

🔴 Negative

A ↑ → B ↓

Пример:

War ↑ → Risk assets ↓

6. Вес (weight)
   Что это:

насколько сильно A влияет на B

Диапазон:

0.0 – 1.0

Практика для MVP:

0.2 – слабое влияние  
0.5 – среднее  
0.8 – сильное

Важно:

Сумма весов для одного target:

≈ 1 (желательно)

(или нормализовать в коде)
7. Пример правильной конфигурации
   Target: BTC grows

Fed cuts → BTC (0.4, positive)  
Liquidity → BTC (0.3, positive)  
Risk-off → BTC (0.3, negative)

8. Ошибки, которые нужно избежать

❌ Слишком много связей (шум)
❌ Случайные связи
❌ Дублирование логики
❌ Отсутствие веса
❌ Непонятное направление
9. Минимальные требования для MVP

Для старта:

5–10 рынков  
10–20 связей максимум

👉 не больше
10. Проверка качества связей

Перед добавлением:

Задайте себе 3 вопроса:

1. Есть ли причинно-следственная логика?
2. Понятно ли направление?
3. Можно ли это объяснить словами?

Если нет — не добавлять
11. Definition of Done

Шаг выполнен, если:

✅ можно добавлять связи
✅ связи хранятся в БД
✅ один рынок имеет несколько факторов
✅ веса и типы учитываются
🔚 Коротко для программиста

    Нужно реализовать систему хранения и управления связями между рынками (source → target) с весами и типами влияния, которые будут использоваться в расчёте expected probability.
